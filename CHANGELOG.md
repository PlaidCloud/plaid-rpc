# Changelog

## Unreleased

- Make `verify_ssl` a real connection option and forward it to the transport ([sc-23168](https://app.shortcut.com/plaidcloud/story/23168)): `Connect.ready()` and `PlaidXLConnect` built `SimpleRPC` without `verify_ssl`, so `SimpleRPC` coerced its own `None` default to `False` and passed it to `requests` as `verify=False`. **TLS certificate verification was therefore off on every RPC call made through `Connect`** — UDFs, Jupyter and server Panel apps — with `urllib3.disable_warnings(InsecureRequestWarning)` suppressing the warning that would have revealed it. `PlaidConfig.verify_ssl` was read from `__PLAID_VERIFY_SSL__` and then dropped on the floor. `PlaidConfig` and `Connect` now accept a keyword-only `verify_ssl`, `Connect.ready()` passes it through, and `PlaidXLConnect` takes one too. **Certificate verification is now on by default from every source** — the `verify_ssl` argument, the `__PLAID_VERIFY_SSL__` environment variable when absent, and the new `verify_ssl` key in `plaid.conf` when absent. **This is a behaviour change for any deployment that has been running unverified without saying so.** Anything addressing an internal or self-signed endpoint must now opt out explicitly with `__PLAID_VERIFY_SSL__='False'` or `verify_ssl: false`. JupyterHub already sets the variable to `'False'` in all 29 tenant values files, and workflow-runner propagates its own `'False'` into every UDF environment, so those two are unaffected ([@inviscid](https://github.com/inviscid)).

- Configure a connection from explicit arguments ([sc-21805](https://app.shortcut.com/plaidcloud/story/21805)): `PlaidConfig` and `Connect` accept keyword-only `rpc_uri`, `auth_token`, `token_provider`, `workspace_uuid` and `project_id`. Passing `rpc_uri` configures from those values alone, skipping both the six-`__PLAID_*`-environment-variable path and the `plaid.conf` file — the only option where neither is available. `token_provider` is a callable resolved per request (`SimpleRPC` already supported this), applies whichever source configured the rest, and is deliberately separate from `auth_token` so that stays a `str` for callers that format it into a header. Existing callers pass no new arguments and take the same path as before. Two supporting fixes: `_C` was a class attribute, so config built in one place leaked into every other instance in the process — it is now per-instance; and `_init_plaidcloud_config` read the raising `project_id` property, so a project-less connection could not be constructed at all — it now reads `_project_id`, leaving the property to raise at the point a project is actually needed ([@inviscid](https://github.com/inviscid)).

- Add `is_dialect_databricks_based` and extend the Plaid* type decorators for the Snowflake/Databricks lakehouse backends ([sc-23158](https://app.shortcut.com/plaidcloud/story/23158)): `PlaidNumeric` → Databricks `DECIMAL(38, 10)`; `PlaidTimestamp` → Databricks `TIMESTAMP_NTZ` (Snowflake rides the plain `TIMESTAMP` impl via session `TIMESTAMP_TYPE_MAPPING`); `PlaidUnicode` → Snowflake `VARCHAR`, Databricks `STRING`; `PlaidTinyInt` → Databricks `TINYINT`; `PlaidJSON` → `VARIANT` on both, with binds serialized to JSON strings wrapped in `PARSE_JSON` and reads parsed back to Python objects (dict-in/dict-out parity with JSONB). Vendor dialects stay lazily imported — nothing new is required at module load ([@inviscid](https://github.com/inviscid)).

- Add a `currency` analyze data type: the new `PlaidCurrency` SQLAlchemy type renders an explicit `DECIMAL(18, 4)` on every warehouse dialect (8-byte Decimal64 values on Databend), with matching pandas (`float64`) and arrow (`decimal128(18, 4)`) conversions. Type inference is unchanged — `currency` source-type spellings still map to `numeric`; only an explicitly stored `currency` dtype gets the narrow type ([@inviscid](https://github.com/inviscid)).
